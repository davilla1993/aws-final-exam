1. Introduction et Contexte

Ce rapport présente la conception et le déploiement d'un pipeline serverless d'ingestion de données IoT en temps réel sur Amazon Web Services, réalisé dans le cadre du cours Introduction à AWS.

Le contexte applicatif est celui d'une entreprise industrielle déployant des milliers de capteurs IoT. L'objectif est de mettre en place une architecture hautement disponible, capable de capter les flux de données envoyés par requêtes HTTP POST, de les stocker dans un Data Lake brut sur S3, et d'alimenter un Feature Store analytique sur DynamoDB.

En parallèle, une documentation technique est hébergée dans un bucket S3 privé, rendu accessible de manière sécurisée via Amazon CloudFront avec le mécanisme OAC (Origin Access Control).

L'intégralité de l'infrastructure est automatisée avec AWS SAM (Serverless Application Model) et déployée via une seule commande sam deploy, sans aucune action manuelle dans la console.

8. Réponses aux Questions Théoriques

Question 1 — Expliquez le concept d'Infrastructure as Code (IaC) et décrivez brièvement le rôle d'AWS CloudFormation dans la gestion d'un cycle de vie applicatif.

L'Infrastructure as Code (IaC) est une pratique qui consiste à décrire et provisionner l'infrastructure informatique via des fichiers de configuration versionnés, au lieu d'actions manuelles dans une console. L'infrastructure devient reproductible, auditable et intégrable dans un pipeline CI/CD.

AWS CloudFormation est le service IaC natif d'AWS. Il interprète un template YAML ou JSON décrivant les ressources souhaitées et orchestre leur création, mise à jour ou suppression en tant qu'unité cohérente appelée stack. Tout au long du cycle de vie applicatif, CloudFormation garantit la cohérence entre l'état décrit et l'état réel des ressources, et permet un rollback automatique en cas d'échec de déploiement.

Dans ce projet, AWS SAM étend CloudFormation avec des abstractions serverless (AWS::Serverless::Function, AWS::Serverless::HttpApi) qui simplifient la déclaration tout en générant les ressources CloudFormation sous-jacentes.

Question 2 — Qu'est-ce qu'une fonction AWS Lambda ? En quoi son approche "Serverless" se distingue-t-elle d'une architecture classique basée sur des instances virtuelles Amazon EC2 ?

AWS Lambda est un service d'exécution de code à la demande. Le développeur fournit uniquement le code (ici Python 3.11) et Lambda se charge de l'exécuter en réponse à un événement, sans qu'aucun serveur ne soit à provisionner, configurer ou maintenir.

CritèreLambda (Serverless)EC2 (classique)
ProvisionnementAutomatiqueManuel
FacturationPar invocation et durée d'exécutionÀ l'heure, serveur allumé ou non
ScalabilitéAutomatique et instantanéeManuelle ou via Auto Scaling
Maintenance OSGérée par AWSÀ la charge du client

Pour un pipeline IoT à charge variable, Lambda est nettement plus adapté qu'EC2 : aucun coût à vide et mise à l'échelle transparente face aux pics d'ingestion.

Question 3 — Quel est l'intérêt architectural d'adosser une distribution Amazon CloudFront devant un point d'entrée Amazon API Gateway pour une collecte globale de données IoT ?

Placer CloudFront devant API Gateway apporte quatre avantages majeurs :

  1. Réduction de la latence : CloudFront dispose de plus de 400 points de présence mondiaux. Les capteurs se connectent au PoP le plus proche, réduisant le temps de transit réseau.
  2. Absorption des pics : CloudFront agit comme bouclier, protégeant API Gateway des surcharges soudaines.
  3. Sécurité renforcée : Intégration native avec AWS WAF et AWS Shield pour filtrer les requêtes malveillantes avant qu'elles n'atteignent le backend.
  4. HTTPS unifié : CloudFront centralise la terminaison TLS avec un certificat géré, indépendamment de la région de déploiement de l'API.

Dans ce projet, la politique CachingDisabled est appliquée car les requêtes POST IoT sont uniques et ne doivent jamais être mises en cache.

Question 4 — Dans un écosystème Big Data, pourquoi choisit-on de stocker les données brutes sur Amazon S3 (Data Lake) et les indicateurs agrégés sur Amazon DynamoDB (Serving Layer) plutôt que de tout centraliser dans une unique base de données relationnelle ?

Séparer les couches de stockage répond au principe de découplage par usage :

Amazon S3 (Data Lake) est optimisé pour stocker des volumes massifs de données brutes non structurées à faible coût. Il supporte des pétaoctets, offre un partitionnement natif par préfixes de clés, et sert de source pour des outils analytiques (Athena, Glue, EMR).

Amazon DynamoDB (Serving Layer) est optimisé pour des lectures/écritures à faible latence sur des données structurées et agrégées. Sa scalabilité horizontale automatique le rend idéal pour alimenter des dashboards en temps réel.

Une base relationnelle unique ne peut pas remplir ces deux rôles simultanément : elle est trop coûteuse à l'échelle pour le stockage brut, et ses mécanismes ACID deviennent un goulot d'étranglement à haute volumétrie.

Question 5 — Expliquez le principe du modèle de responsabilité partagée d'AWS concernant la sécurité des données stockées dans Amazon S3.

Le modèle de responsabilité partagée définit une frontière claire entre ce qu'AWS gère et ce que le client doit sécuriser.

AWS est responsable de la sécurité physique des datacenters, de la durabilité de l'infrastructure de stockage (11 neuf de durabilité), et du chiffrement de l'infrastructure sous-jacente.

Le client est responsable de la configuration des politiques d'accès (Bucket Policy, IAM), de l'activation du chiffrement des objets (SSE-S3, SSE-KMS), du blocage des accès publics, et de la conformité réglementaire des données stockées.

Dans ce projet, le client applique ce modèle en activant BlockPublicAccess sur les deux buckets et en restreignant les accès via IAM (rôle Lambda) et OAC (CloudFront).

Question 6 — Pourquoi est-il fortement déconseillé d'activer le "Static Website Hosting" public sur un bucket S3 pour héberger une documentation interne ? En quoi l'utilisation d'une distribution CloudFront combinée avec un OAC améliore-t-elle la sécurité ?

Activer le Static Website Hosting public sur S3 expose le bucket directement sur Internet via une URL S3 prévisible. Cela implique que tous les objets doivent être publics, que la surface d'attaque est maximale, qu'aucun contrôle d'accès fin n'est possible, et qu'aucun HTTPS natif n'est disponible sur l'endpoint S3 website.

CloudFront + OAC résout ces problèmes : le bucket reste entièrement privé (Block Public Access activé) et CloudFront est le seul autorisé à lire le bucket, via une signature SigV4 vérifiée par S3 grâce à une condition AWS:SourceArn sur l'ARN exact de la distribution.

La validation dans ce projet est concrète : l'accès direct S3 retourne 403 Access Denied, tandis que l'URL CloudFront (https://d3l8f210waz845.cloudfront.net) affiche correctement la documentation.

Question 7 — Comment Amazon CloudWatch permet-il de superviser et de déboguer le code d'une application serverless ? Que se passe-t-il au niveau des logs si la fonction Lambda lève une exception non gérée ?

Amazon CloudWatch collecte automatiquement les logs de chaque invocation Lambda dans un Log Group dédié (/aws/lambda/fgbossou-iot-pipeline-ingestion). Chaque exécution crée un Log Stream horodaté contenant les messages émis via logger.info() / logger.error() et le rapport d'exécution AWS (durée, mémoire consommée, statut).

En cas d'exception non gérée, Lambda intercepte automatiquement l'erreur et écrit dans CloudWatch : le message d'exception, la stack trace Python complète (fichier, numéro de ligne, type d'erreur), et le statut ERROR dans le rapport d'exécution.

Dans ce projet, les tests avec payload corrompu ont généré exactement ces traces, visibles dans les captures 15 et 16 ci-dessus.

Question 8 — Imaginons qu'un fichier de données de 50 Go soit déposé d'un coup. Pourquoi l'architecture AWS Lambda actuelle va-t-elle atteindre ses limites ? Quel service managé d'AWS orienté Big Data devrions-nous utiliser ?

AWS Lambda présente des limites physiques incompatibles avec un fichier de 50 Go :

Limite LambdaValeur
Durée maximale d'exécution15 minutes
Mémoire maximale10 240 MB
Taille du payload entrant (synchrone)6 MB
Stockage temporaire /tmp10 GB

Un fichier de 50 Go ne peut pas être chargé en mémoire Lambda, ne peut pas être stocké dans /tmp, et son traitement dépasserait largement les 15 minutes autorisées.

Service recommandé : AWS Glue. C'est le service ETL managé d'AWS qui exécute des jobs Apache Spark serverless sur des données stockées dans S3, sans infrastructure à gérer, et qui scale automatiquement sur des téraoctets. Il s'intègrerait naturellement dans notre architecture en lisant depuis le Data Lake S3 (raw-zone/) pour produire des agrégats complexes dans DynamoDB ou Amazon Redshift.

9. Conclusion

Ce projet démontre la faisabilité d'un pipeline d'ingestion IoT entièrement serverless sur AWS, déployé via une seule commande SAM sans aucune action manuelle.

L'architecture mise en place répond à toutes les exigences du cahier des charges : collecte temps réel via CloudFront et API Gateway, traitement et validation des données par Lambda Python 3.11, persistance dans un Data Lake S3 partitionné et un Feature Store DynamoDB, et documentation technique sécurisée via CloudFront OAC.

Les tests de bout en bout confirment le bon fonctionnement de chaque composant : HTTP 201 pour les payloads valides, HTTP 400 avec messages explicites pour les erreurs de validation, logs CloudWatch structurés pour le débogage, et blocage effectif des accès directs S3.

Cette architecture, bien qu'orientée examen, est directement transposable en production. Pour un passage à l'échelle industrielle, on envisagerait l'ajout d'Amazon Kinesis Data Streams en amont de Lambda pour absorber les pics massifs, et AWS Glue pour le traitement batch des gros volumes.
